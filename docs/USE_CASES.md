# MADO queue — ユースケースと現場の使い方

> このドキュメントは、**初めて MADO queue を触る人**が「これは何のためのシステムか」を把握するための文書です。
> 機能要件の一覧は [REQUIREMENTS.md](REQUIREMENTS.md)、API・DB・印刷の詳細は [ARCHITECTURE.md](ARCHITECTURE.md) を参照してください。

**対象読者:** 導入を検討する自治体の担当者、コントリビューター、フォークしてカスタマイズする開発者

---

## 1. ひとことで言うと

**MADO queue は、住民窓口の来庁者に番号札を渡し、職員が呼び出し・完了を管理し、待合室に番号を表示する、小規模自治体向けの番号発券システムです。**

来庁者が「何の用件で来たか」をボタンで申告し、窓口の混雑を可視化し、待ち時間・処理時間を記録します。個人情報は扱わず、受付用 LAN 上で単体動作します。

MADO 全体（申請書自動転記など）の入口として設計されていますが、**queue 単体でも番号管理システムとして独立して使えます。**

## 2. 誰が、いつ、何のために使うか

### 2.1 登場人物と画面

| 誰 | 使う画面 | いつ | 何のために |
|---|---|---|---|
| **来庁者（市民）** | 番号札 | 受付時 | 順番待ちの目印を手に入れる |
| **総合案内・受付職員** | `/` 発券画面 | 来庁直後 | 用件を確認し、適切なカテゴリの番号を発行する |
| **窓口職員** | `/processing` 処理画面 | 席にいる間ずっと | 待ち一覧を見て呼び出し・完了を記録する |
| **待合室の来庁者** | `/display` 案内表示 | 待っている間 | 自分の番号が呼ばれたかを目と耳で確認する |
| **窓口管理者** | DB（`event_logs` / `processing_logs`） | 日次・月次 | 混雑傾向・待ち時間・職員配置との関係を把握する |

> **芽室町での運用:** 入口に総合案内を置き、**職員がタブレットを操作して発券**しています（来庁者が自分でタッチする方式ではありません）。各自治体の運用に合わせてどちらでも構いません。→ [§7 フォーク先向け](#7-フォーク先向けボタン設計の考え方)

### 2.2 解決したい困りごと

| 導入前（よくある状態） | 導入後 |
|---|---|
| 口頭で「次の方」と案内し、来庁者が聞き逃す | 大画面＋チャイムで視覚・聴覚の両方から通知 |
| 紙の番号札では「何の用件か」が窓口に伝わらない | 発券時に手続き種別を記録し、処理画面に表示 |
| 混雑状況は窓口に行かないと分からない | 職員が自席から `/processing` で待ち人数を確認 |
| 待ち時間・処理時間のデータが残らない | 発券・呼び出し・完了のタイムスタンプを DB に蓄積 |
| 高機能な番号管理 SaaS はコスト・カスタマイズが重い | OSS・SQLite・Docker で小規模自治体が自前運用可能 |

## 3. 典型シナリオ（1日の流れ）

### シナリオ A：平日午前の通常運用

**状況:** 8:30 開庁。窓口職員 5 名、総合案内 1 名。

```
08:25  総合案内職員が発券タブレットを開き、「対応職員数：5」を選択
08:32  来庁者A「住民票が欲しい」→ 職員が「住民票」ボタン → 番号 001 発行
08:35  来庁者B「転入届」→ 「転入」ボタン → 番号 502（カテゴリ B）
08:40  窓口職員が自席 PC で /processing を開く
08:41  001 を「呼び出し（処理開始）」→ /display に大きく表示＋チャイム
08:55  手続き完了 → 「完了（処理終了）」→ 待ち時間・処理時間が DB に記録
```

**ポイント:**
- 番号の **3 桁目でカテゴリが一目で分かる**（001 = A、502 = B、801 = C）
- 職員は窓口を離れずに、自席から次の来庁者を呼べる
- `/display` は「聞こえなかった」「窓口がどこか分からない」来庁者の安全網

### シナリオ B：混雑時（月曜午前・月初）

**状況:** 待ち 15 人。カテゴリ A が集中。

```
09:10  /display に「お待ちの方 15 人」と表示
09:12  窓口職員が A 系を優先的に呼び出し（短時間手続きを先に捌く）
09:15  B 系の来庁者には番号札に「カテゴリAの方を先にご案内する場合があります」と印字
09:20  不在の来庁者 → 「不在（待ちに戻す）」で待ち列に復帰
09:25  取り消し・間違い発券 → 「削除」で履歴に残しつつ待ち列から除外
```

**ポイント:**
- カテゴリ A/B/C は **別窓口に固定割り当てではなく**、必要な技能・所要時間の目安
- B/C の番号札注記は、A を優先する運用があることを来庁者に事前告知するため
- 「不在」「削除」は現場で毎日起きる操作として用意されている

### シナリオ C：段階導入（スモールスタート）

**状況:** まずは紙の番号札に替わる発券だけ試したい。

```
Phase 1  タブレット + レシートプリンター + / のみ導入
         → 来庁者に番号と用件が印刷されたチケットを渡せる
Phase 2  /processing を職員 PC に追加
         → 待ち時間・処理時間のログが取れる。離席中も混雑把握
Phase 3  /display を待合室モニターに追加
         → 音声案内だけでは足りない来庁者への視覚的通知
```

**ポイント:**
- Phase 1 だけでも「用件付き番号札」として機能する
- Phase 2・3 は **後から追加しても DB は共通**（芽室町もこの順で拡張した）

## 4. 設計の「なぜ」を読み解く

画面を初見で触ると意味が分かりにくい部分があります。現場の意図を言語化します。

### 4.1 カテゴリ A / B / C / D とは

| カテゴリ | 番号帯 | 芽室町での意味 | 来庁者への見え方 |
|---------|--------|--------------|----------------|
| **A** | 001–499 | 証明書など比較的短時間・初級窓口向け | 青ボタン |
| **B** | 500–799 | 異動・登録・マイナンバーなど中級向け | グレーボタン |
| **C** | 800– | 戸籍届出・おくやみなど正職員・専門向け | 緑ボタン |
| **D** | 連番のみ | 窓口手続き以外の来庁（見学・配布物など） | 印刷なし |

**なぜ番号帯を分けるか:**
- 番号だけ見ても用件の重さが推測できる（545 なら B 系）
- カテゴリごとに独立カウンター → 日次リセットが単純
- 処理画面・表示画面でカテゴリ別に扱いやすい

**誤解しやすい点:** カテゴリ = 物理的な窓口番号、ではありません。あくまで **手続きの種類と技能レベルの分類** です。窓口の割り当ては当日の職員配置で柔軟に変わります。

### 4.2 「対応職員数 1〜7」とは

発券画面の上部にある、**その時点で窓口に対応できる職員数**の申告です。

**誰が操作するか:** 芽室町では **総合案内の職員** が、開庁時や昼休み明けなど配置が変わるタイミングで選択します。

**何のために記録するか:**
- 発券ログ（`event_logs.staff_count`）と処理ログ（`processing_logs.staff_count`）に紐づく
- 「職員 3 人の日は平均待ち 12 分、6 人の日は 5 分」のような **配置と待ち時間の相関分析** に使う
- システムが職員数を自動検知する機能はない（現場の申告ベース）

**任意項目:** 未選択でも発券は可能です。分析精度のため、運用上は選択を推奨します。

### 4.3 なぜ 2 枚印刷するか

1 回の発券で同一内容のチケットを **2 枚** 連続印刷します。

| 枚 | 想定される用途 |
|---|---|
| 1 枚目 | 来庁者が持つ番号札 |
| 2 枚目 | 総合案内・職員控え、または来庁者の予備 |

物理的な受け渡しルールは自治体ごとに異なります。2 枚にしている理由はコード上には明文化されていませんが、**来庁者用と窓口側用の二部構成**を想定した運用と考えられます。

### 4.4 3 つの画面が分かれている理由

| 画面 | 分離する理由 |
|-----|------------|
| `/` | タブレット・タッチ操作・プリンター接続。来庁者（または案内職員）が触る |
| `/processing` | 職員専用操作（削除・不在・完了）。誤操作防止のため URL で分離 |
| `/display` | 大画面・自動更新・チャイム。操作不要のサイネージ専用 |

URL が違うだけで、**同じ DB を見ている** ので、発券 → 呼び出し → 表示はリアルタイムに連動します。

## 5. 記録されるデータと、その使い道

queue は BI ダッシュボードを内蔵していません。SQLite にログを蓄え、**各自治体が必要に応じて分析** する前提です。

### 5.1 何が記録されるか

| タイミング | 記録内容 | テーブル |
|-----------|---------|---------|
| 発券 | カテゴリ、手続き名、番号、発券時刻、職員数 | `event_logs` |
| 呼び出し | 待ち時間（分）、呼び出し時刻 | `processing_logs` |
| 完了 | 処理時間（秒）、完了時刻 | `processing_logs` |
| 削除 | 削除ステータスで履歴保持 | `processing_logs` |

### 5.2 現場での活用例

- **手続き別の平均待ち時間** — どのボタン（用件）に時間がかかるか
- **時間帯別の混雑** — 午前集中か、昼休み明けか
- **職員数と待ち時間の関係** — 配置計画の根拠データ
- **月初・年度末の来庁ピーク** — 臨時窓口増設の判断材料

> 分析用 SQL やダッシュボードは現時点では同梱されていません。必要に応じてフォーク先で追加するか、Issue で議論してください。

## 6. 向いている自治体・向いていないケース

### 向いている

- 人口〜5 万人程度の小規模自治体
- 住民窓口の来庁者数が 1 日 50〜200 件程度
- 庁内に受付 LAN があり、タブレット・PC・モニターを置ける
- 手続き種別をボタンで自己申告（または案内職員が代行）できる運用
- OSS をフォークして各自治体の窓口業務に合わせたい

### 向いていない・別手段を検討すべき

- 大規模自治体で全庁舎・全課を横断管理したい → 中・大規模向け SaaS の方が適する場合がある
- 予約制・オンライン事前申込が主流で、当日 walk-in が少ない
- 個人情報と連携した本人確認付き受付が必要（queue は個人情報非保持）
- 音声案内・多窓口振り分け・SLA アラートなど高度な機能が必須

## 7. フォーク先向け：ボタン設計の考え方

`templates/index.html` のボタン一覧は **芽室町の窓口業務そのもの** です。そのまま使うのではなく、各自治体の業務に合わせて設計し直してください。

### 7.1 ボタンを決める手順

1. **窓口で受け付ける手続きを洗い出す**（職員ヒアリング）
2. **所要時間・必要技能で 3 グループに分類** → カテゴリ A/B/C に対応
3. **来庁者が選びやすい名前** にする（行政用語より市民向け表現）
4. **1 ボタン = 1 用件** を基本に。似た手続きは統合してよい
5. **窓口以外の来庁** があればカテゴリ D を使う

### 7.2 カテゴリ分けの目安

| カテゴリ | 分類基準（例） |
|---------|--------------|
| A | 5〜10 分程度、パート・初級職員でも対応可能 |
| B | 15〜30 分、専門知識または特定システム操作が必要 |
| C | 30 分以上または正職員・管理職の判断が必要 |
| D | 窓口手続きではないが来庁者数の把握が必要 |

### 7.3 発券方式の選択

| 方式 | 説明 | 変更の目安 |
|-----|------|-----------|
| **職員発券**（芽室町） | 総合案内が来庁者の用件を聞いてボタンを押す | 現状のまま。職員数選択も案内職員が操作 |
| **来庁者自助発券** | 来庁者が自分でタッチして番号を取る | UI 文言・ボタンサイズ・案内表示の追加が必要な場合あり |

→ 発券方式の変更は [CONTRIBUTING.md](../CONTRIBUTING.md) にも例として記載されています。設計上の検討は Issue で共有歓迎です。

## 8. MADO 全体との位置づけ

```
来庁 ──→ [queue] 番号発券・呼び出し  ← このリポジトリ（独立動作）
              │
              ↓ （将来・別リポジトリ）
         [hub]  住民情報
              │
         ┌────┼────┐
         ↓    ↓    ↓
      [form] [care] [move]  申請書・手続き支援
```

queue は **個人情報を扱わない** ため、受付 LAN だけで完結します。MADO の form（申請書自動転記）と組み合わせる場合、queue で番号を取った後に別システムへ誘導する、という連携イメージです。
