# MADO queue — Requirements & Overview

> 番号発券パッケージ `queue` の目的と業務要件をまとめた概要ドキュメント。

**対象拠点:** 住民窓口
**ステータス:** 北海道芽室町の窓口で本番稼働中

> 導入を検討する自治体・コントリビューターが、システムの目的と業務上の要件を把握するための文書です。
> **「現場でどう使うか」のユースケース・シナリオ** は [USE_CASES.md](USE_CASES.md) を先に読むと分かりやすい。実装レベルの詳細は [ARCHITECTURE.md](ARCHITECTURE.md) を参照してください。

---

## 1. プロジェクト概要

### 1.1 背景・目的

住民窓口における来庁者の待ち時間管理と受付業務の効率化を目的とし、番号発券・呼び出し・処理管理を一元的に行うシステムを構築する。

市民が窓口へ来庁した際に番号札を発行し、職員が呼び出し操作を行い、大型ディスプレイに呼び出し番号を表示することで、来庁者の待ち時間体験を改善するとともに、窓口業務のデータを収集する。

「速く処理する」ことよりも、システム導入により生まれた余力を「丁寧に寄り添う」窓口応対へ充てることを優先した設計とする。

### 1.2 対象ユーザー

| ユーザー種別 | 利用画面 | 説明 |
|------------|---------|------|
| 来庁者（市民） | 発券画面 | 番号札を取得する一般市民 |
| 窓口職員 | 処理管理画面 | 呼び出し・処理完了を操作する担当職員 |
| 待合室の来庁者 | 表示画面 | 大型ディスプレイで呼び出し番号を確認する来庁者 |

## 2. システム要件

### 2.1 機能要件

#### FR-001: 番号発券機能
- 来庁者が手続き種別を選択すると、カテゴリごとに連番の番号が発行される
- 発行された番号は物理チケットとしてレシートプリンターから印刷される（一部カテゴリを除く）
- チケットには手続き種別、日時、番号を印字する
- 番号は毎日0時に自動リセットされる

#### FR-002: カテゴリ管理機能
- 手続きの種類に応じて以下の4カテゴリに分類する
  - **カテゴリ A**（証明書系・比較的短時間）: 番号 1〜499
  - **カテゴリ B**（異動・登録系）: 番号 500〜799
  - **カテゴリ C**（特殊手続き系）: 番号 800〜
  - **カテゴリ D**（窓口以外の来庁者）: 番号発行のみ（印刷なし）

#### FR-003: 呼び出し管理機能
- 窓口職員は待ち一覧から番号を選択して「呼び出し（処理開始）」操作ができる
- 呼び出し中のチケットは「対応中」として区別される
- 来庁者が不在の場合、チケットを「待ち」状態に戻すことができる
- 処理完了後は「完了（処理終了）」操作により履歴として記録される

#### FR-004: 公開表示機能
- 大型ディスプレイに呼び出し中の番号と待ち人数をリアルタイム表示する
- 新しい呼び出しが発生した際に音声チャイムを鳴らす
- 3秒ごとに自動更新する

#### FR-005: データ記録・分析機能
- チケット発行ログ（手続き種別、発行時刻、番号）を記録する
- 処理ログ（待ち時間、処理時間）を記録する
- 日次での番号リセット時に当日データを保持する

#### FR-006: 削除・キャンセル機能
- 不要なチケットを待ち一覧から削除できる
- 削除したチケットは「deleted」ステータスで履歴に保持する

### 2.2 非機能要件

#### NFR-001: パフォーマンス
- 番号発行レスポンスタイム: 2秒以内
- 表示画面の更新間隔: 3秒
- 管理画面の自動更新間隔: 10秒

#### NFR-002: 可用性
- 庁舎開庁時間帯（平日 8:00〜17:30）に安定稼働すること
- サーバー再起動後も当日のデータが保持されること

#### NFR-003: 操作性
- 来庁者向け発券画面はタブレット端末で操作可能であること
- 職員向け管理画面はPCブラウザで操作可能であること
- 表示画面は32インチ以上の大型モニターに対応すること

#### NFR-004: 印刷
- USB接続のレシートプリンター（ESC/POS対応機種）により印刷すること
- チケット発行ごとに2枚印刷すること
- カテゴリDは印刷不要

#### NFR-005: セキュリティ
- 受付ネットワーク上での独立運用を前提とし、庁内システムとのネットワーク接続は不要
- 住民の個人情報を保持しない設計とする
- 来庁者向け画面と職員向け画面はURLで分離する

#### NFR-006: 保守性
- SQLiteを使用し、外部DBサーバーなしで単体動作すること
- Docker による起動を標準とし、Windowsローカル環境でも直接起動できること

## 3. 業務フロー

```
[来庁者]
  ↓ タブレット端末で手続き種別を選択
[発券画面]
  ↓ 番号発行・チケット印刷（USB接続のレシートプリンターへ送信）
[来庁者が番号札を受け取り、待合エリアへ]
  ↓
[待合エリアの大型ディスプレイで呼び出し番号を確認]
  ↑ 3秒ごと更新
[表示画面]
  ↑ /display_data API
[職員が処理管理画面で「呼び出し」操作]
  ↓
[来庁者が窓口へ]
  ↓
[手続き完了後、職員が「処理完了」操作]
  ↓
[データがDBに記録（待ち時間・処理時間）]
```

## 5. 制約事項

- 印刷機能は ESC/POS 対応のレシートプリンターを USB 接続で使用する
- 印刷の実行環境は次のいずれか
  - **Windows**: `win32print`（pywin32）経由でWindowsプリンターへRAW送信
  - **Linux / Docker**: `pyusb` 経由でUSBデバイスへ直接送信
- LAN（受付ネットワーク）環境内での運用を前提とする（複数端末からのアクセス可）
- ブラウザはモダンブラウザ（Chrome / Edge 推奨）を使用すること
- 発券用タブレットとプリンターは同一サーバーに接続して設置すること

---

## 6. 前提条件

- 庁舎内のLAN（受付ネットワーク）が整備されていること
- Docker が利用できる環境、または Python および依存パッケージがインストールされたサーバーがあること
- ESC/POS 対応レシートプリンターが設置され、サーバーにUSB接続されていること
- プリンター機種に応じて `PRINTER_NAME` / `PRINTER_VID` / `PRINTER_PID` を設定できること（詳細は [ARCHITECTURE.md](ARCHITECTURE.md) 参照）
- 大型ディスプレイが待合エリアに設置され、ネットワーク接続されたPCで表示されること
